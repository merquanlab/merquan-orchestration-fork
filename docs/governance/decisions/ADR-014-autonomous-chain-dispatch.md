# ADR-014 — Autonomous Mode = Pre-Approved Chain Dispatch, Never Gate Bypass

**Status:** Accepted
**Date:** 2026-05-09
**Decided by:** Operator (Vincent van Deth)
**Resolves:** Wave 0.5 prerequisite + PRD-VNX-UH-002 §6 FR-13 + tension between operator's autonomous-mode preference and ADR-006's mandatory human approval gate
**Cross-references:** ADR-005 (NDJSON ledger), ADR-006 (staging→promote human gate), ADR-008 (dual-LLM review)

## Context

ADR-006 mandates that every VNX dispatch passes through a human-operator promote action. The pending→active transition is the explicit consent boundary, and ADR-006 is unambiguous: "There is NO auto-execute path … There is NO autonomous-mode bypass. There is NO low-risk-dispatch shortcut."

Simultaneously, the operator runs long-running autonomous workflows. The most concrete examples:

1. **F60 / overnight feature work.** The operator dispatches a feature kickoff at end-of-day, sleeps, and expects 5–15 dispatches to have run by morning — codex gate, gemini review, fix-forward iterations, CI re-run, merge. Sitting in front of the screen and pressing promote 15 times across an 8-hour window is not the workflow.
2. **The `/loop` skill** (registered in `.claude/skills/loop`) explicitly exists "to run a prompt or slash command on a recurring interval … keep running /babysit-prs." Its semantics are inherently autonomous-multi-dispatch.
3. **The "Auto Mode Active" pattern in the project root `CLAUDE.md`.** Auto mode says: "Execute immediately … Make reasonable assumptions and proceed on low-risk work … Prefer action over planning." Auto mode and per-dispatch promote are obviously in tension.
4. **The `t0-orchestrator` skill chain semantics.** T0's CLAUDE.md (`/Users/vincentvandeth/Development/vnx-roadmap-autopilot-wt/.claude/terminals/T0/CLAUDE.md`) instructs: "for the next 4-feature hardening lane, operate in full autonomous mode: no routine user checkpoints … after each feature: close → merge → verify merge → create next branch/worktree." That instruction routinely covers 20+ underlying dispatches.

ADR-006 already names the resolution in its `Accepted` list: "Autonomous chain pre-approval is the only operator-level relaxation. The operator may declare 'I pre-approve all dispatches in this chain that match this scope' at chain kickoff." But ADR-006 is principle-level — it does not specify what a chain *is*, what it must declare, what its termination criteria are, how the consent is recorded in the audit ledger, or what happens when the chain wants to fork outside its approved scope.

That underspecification is the gap PRD-VNX-UH-002 §6 FR-13 calls out and gates on a Wave 0.5 ADR. This ADR is that ADR.

## Decision

**Autonomous mode is a chain-dispatch primitive.** The operator pre-approves a *chain* — a bounded sequence of dispatches with explicit scope, termination criteria, and escalation rules. Within the approved chain, T0 may dispatch sequential or parallel steps without per-step promote action. The chain itself is the unit of human approval. Each dispatch within the chain still flows through `pending/` → `active/` filesystem mechanics; the promote step is performed by a chain-runner daemon that consumes the operator's pre-approval signature, not by the operator's hand. The promote step's *cause* changes (signed chain spec instead of live keystroke); the gate's *existence* does not.

A chain MUST declare, at the moment of the operator's pre-approval action:

- **Scope** — file paths, repos, branches, and dispatch types the chain may touch. Any dispatch generated by T0 that falls outside the declared scope is automatically deflected back to interactive single-dispatch promote.
- **Termination criteria** — explicit success state ("all 4 features merged + main green"), blocking-finding rules ("any codex blocking finding terminates the chain pending operator review"), max-step count ("≤30 dispatches"), max-cost ("≤$15 in worker tokens"), and max-wallclock ("≤12 hours").
- **Escalation triggers** — conditions under which the chain pauses and wakes the operator. Default escalation set per the operator policies in T0's CLAUDE.md (E1–E6: main broken, data-loss risk, secrets in logs, GitHub auth dead, codex >5h unavailable, three consecutive same-issue blockers).
- **Reverse-out behavior** — what happens when termination fires mid-chain: leave dispatches in active/, close them, revert merges, file open items? Default: in-flight dispatch is allowed to finish-with-receipt, queued dispatches stay in `pending/` for operator review, no merges are reverted automatically.
- **Chain spec hash** — the chain spec is canonicalized and SHA-256-hashed at approval time. The hash is appended to the NDJSON ledger as the consent token. Every promote performed under the chain references the hash.

Chains are recorded in `.vnx-data/chains/` with three states (`approved/`, `running/`, `terminated/`). The chain-runner daemon reads the operator's pre-approval, validates each generated dispatch against the chain's scope, and performs the `pending/` → `active/` move under the chain hash. NDJSON events tag each promote with the chain id and chain hash, making the audit trail queryable per chain.

## Reasoning

1. **ADR-006 mandates a human gate per *unit of operator consent*; the existing pattern made the unit a single dispatch. Chain-dispatch redefines the unit to a *bounded sequence*.** The principle is preserved: the operator consents once, with bounded scope, to a sequence of work — instead of consenting separately to each step. ADR-006 explicitly contemplates this in its Accepted list ("Autonomous chain pre-approval is the only operator-level relaxation"). This ADR specifies the contract.

2. **The operator's `/loop` and "Auto Mode" already do this informally.** Today the operator says "kick off F60 overnight" and a long sequence of dispatches happens, with implicit consent covering the whole sequence. Without formalization that consent is undocumented — it lives only in the operator's head and the orchestrator's decision rules. Formalizing makes the consent auditable, replayable, and challengeable. The chain spec is the artifact that makes "the operator approved this run" something an auditor can read after the fact.

3. **Without chain-dispatch, autonomous mode is either a slippery slope or impossible.** Two failure modes:
   - **Slippery slope.** "Just don't gate the autonomous loop" — leads to AI-to-AI handoff (a worker's follow-up dispatch auto-executes; T0's planned next step auto-executes). ADR-006 forbids this categorically. Without a formal chain primitive, every "autonomous mode" patch tends to slide here.
   - **Impossible.** "Promote every dispatch by hand even in autonomous mode" — kills the F46–F58 + 200-PR cadence the operator actually achieves. The operator runs four parallel deployments; per-step approval is not workable at that throughput.
   Chain-dispatch is the only shape that preserves the M2 moat (mandatory human approval per `claudedocs/2026-05-09-vnx-strategic-replan-proposal.md` §4) while enabling the workflow operators need. It is the smallest viable resolution.

4. **The audit ledger's value depends on consent being explicit, not implicit.** Per ADR-005, NDJSON entries record every dispatch's lifecycle. Per ADR-006, the consent encoding is the `pending/` → `active/` transition. If autonomous mode silently skipped the gate, the ledger would record "the system did this" without the matching "the operator authorized this." Chain-dispatch keeps the consent encoding intact: the chain spec is the consent, the chain hash is its cryptographic anchor, every promote within the chain references the hash, and the operator's signature on the chain spec is auditable for years afterwards.

5. **Bounded scope makes runaway loops impossible.** AI agents that plan their own next steps without hard scope walls exhibit known failure modes — recursive self-dispatch, refinement-loop divergence, scope creep into adjacent files. The chain spec's `scope` field is the wall. A T0 that wants to dispatch outside the wall is automatically deflected back to interactive promote. Termination criteria add a second wall — even within scope, the chain is bounded by step count, cost, and wallclock.

6. **Chain spec replay enables post-incident audit.** Because the chain hash is in the NDJSON ledger and the canonicalized chain spec is in `.vnx-data/chains/approved/`, an auditor (or the operator on Monday morning reviewing Sunday's run) can replay: "what did I approve, what did the system do under that approval, did anything happen outside scope?" The replay is the dual property to ADR-006's "operator can stop something from happening" — here, "operator can prove what they did and didn't authorize."

7. **Aligns with ADR-008's adversarial review.** Dual-LLM review (codex_gate + gemini_review) per ADR-008 must still run on every PR within the chain. The chain spec declares whether review gates run per-PR (the default) or only at chain completion (rare, opt-in for low-risk doc-only chains). The codex/gemini gate posture is independent of the chain consent posture.

## Consequences

### Accepted

- New `chain_dispatch` primitive lands in `scripts/lib/` (new module `chain_dispatch.py`, optionally extending `dispatch_register.py`). The primitive owns chain validation, scope enforcement, termination evaluation, and ledger writes.
- A new operator action — `vnx chain approve <chain-spec-path>` — performs the pre-approval: canonicalizes the spec, computes the SHA-256, signs it with the operator key, writes the spec to `.vnx-data/chains/approved/<chain-id>.json`, and emits a `chain_approved` NDJSON event.
- The chain-runner daemon (a thin wrapper around the existing dispatcher) reads `chains/approved/`, performs validated `pending/` → `active/` promotes under the chain hash, and emits `chain_promote` NDJSON events tagged with the chain id and hash.
- Each dispatch's NDJSON entry gains a `chain_id` and `chain_hash` field when the dispatch was promoted under a chain. The fields are absent for interactive single-dispatch promotes (preserving the existing schema for the dominant non-autonomous case).
- The chain spec is signed/hashed in NDJSON for replay-audit per ADR-005's append-only ledger contract. The spec itself is stored on disk under `.vnx-data/chains/`; the hash is the only thing in the ledger.
- T0 may dispatch within the chain without operator action, **but only when the dispatch's scope matches the chain spec.** Out-of-scope dispatches (different repo, different file paths, different dispatch type) are written to `pending/` and wait for interactive promote, exactly like a non-autonomous dispatch.
- Operator wake-up triggers (E1–E6 in T0's CLAUDE.md, plus the chain spec's `escalation_triggers`) are evaluated at every chain step. When any trigger fires, the chain transitions to `terminated/` and the operator is notified via the daily digest channel and (if configured) email.
- Chain audit appends to the NDJSON ledger continuously; `claudedocs/2026-05-09-vnx-strategic-replan-proposal.md` Wave 0.5 lists the chain-runner observability work as a tracked deliverable.
- `dispatcher_supervisor.sh` and the lease-sweep ticker are extended to detect chain-runner drift (e.g. a chain that has been `running/` for longer than its `max_wallclock` without a terminate event).

### Rejected

- **Dispatch-level autonomous bypass.** "Set a flag on a dispatch to skip the gate" — rejected. The bypass is at the chain level, never at the dispatch level. A dispatch carries no authority to bypass anything.
- **"Yolo mode" without chain bounds.** A free-running autonomous mode without scope, termination, or escalation declarations is rejected. Autonomous mode requires bounded consent, full stop.
- **AI-to-AI dispatch creation without operator-approved chain context.** A worker that generates a follow-up dispatch outside any chain — that follow-up still lands in `pending/` and waits for interactive promote, exactly as ADR-006 requires.
- **Chains that can fork outside their declared scope.** A chain's scope is final at approval time. T0 may not amend the scope mid-chain. If new scope is needed, the operator approves a successor chain. (This is the dual of "can the chain spec be edited while running?" — no, it is immutable; that immutability is what makes the hash a meaningful audit anchor.)
- **Chain auto-approval based on heuristics.** "If the chain looks like a routine refactor, auto-approve it." Rejected — the operator's signature is required for every chain, period. Heuristic auto-approval would re-introduce the gate-skip ADR-006 forbids.
- **Indefinite chains.** Every chain has a `max_step_count`, `max_cost`, and `max_wallclock`. A chain that wants to run forever is rejected at approval time. (This forecloses the "set up an autonomous chain at project start, never re-approve" anti-pattern.)
- **Per-dispatch promote bypass while a chain is running.** When a chain is running, dispatches within scope are auto-promoted under the chain hash; out-of-scope dispatches still require interactive promote. There is no third option.

## Implementation note

- Chain spec format is JSON, schema lives in `schemas/chain_spec.schema.json`. Canonicalization uses the same JCS-style normalization used elsewhere in VNX for `contract_hash` (per ADR-008).
- Operator signature is HMAC-SHA-256 over the canonicalized spec, keyed off `$VNX_OPERATOR_KEY` (a local file under `.vnx-data/state/`, gitignored, generated at `vnx init`). Future ADR may upgrade to ed25519 for multi-operator setups; HMAC suffices for single-operator local-first.
- The chain-runner daemon is a thin extension of the existing dispatcher; it does not introduce a new always-on process beyond what `dispatcher_supervisor.sh` already supervises.
- The 4-feature hardening lane currently described in T0's CLAUDE.md ("operate in full autonomous mode … no routine user checkpoints") is the canonical first consumer. Once chain-dispatch lands, that prose is replaced by an actual `chain_spec` example committed to the repo.
- Wave 0.5 in the strategic replan is the wave that lands this ADR's implementation. Wave 0.5 is gated on this ADR (per FR-13 in PRD-VNX-UH-002) and unblocks the autonomous-mode workflow the operator already wants.
